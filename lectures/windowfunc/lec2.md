# Лекция: Ранжирование данных со смещением через оконные функции в PostgreSQL

## Введение

В предыдущей теме мы рассмотрели общую теорию оконных функций. Однако на практике часто возникает задача не просто вычислить агрегат в разрезе группы (партиции), а **сравнить текущую запись с соседними** или получить доступ к первому/последнему элементу группы без использования группировки (GROUP BY).

Именно здесь на сцену выходят функции **смещения (offset functions)** и **функции доступа к позиции**: `LAG`, `LEAD`, `FIRST_VALUE`, `LAST_VALUE`, `NTH_VALUE`.

В отличие от функций ранжирования (`ROW_NUMBER`, `RANK`, `DENSE_RANK`), которые присваивают номер позиции, эти функции возвращают *фактические данные* из строк, находящихся в определенной позиции относительно текущей строки или в пределах всей партиции.

### Что такое фрейм (frame)

Прежде чем перейти к детальному рассмотрению функций, важно понять концепцию **фрейма** (frame), которая играет ключевую роль в работе некоторых оконных функций.

**Фрейм** — это набор строк внутри партиции, который определяет, какие строки будут включены в вычисление оконной функции для текущей строки. Если партиция — это группа строк (например, все заказы одного пользователя), то фрейм — это подмножество этих строк, которое "видит" функция при обработке конкретной строки.

Фрейм задается с помощью конструкции:
```sql
ROWS | RANGE BETWEEN start AND end
```

Где:
- `ROWS` — фрейм строится на основе физических номеров строк
- `RANGE` — фрейм строится на основе значений в столбце сортировки
- `start` и `end` могут быть:
  - `UNBOUNDED PRECEDING` — от начала партиции
  - `UNBOUNDED FOLLOWING` — до конца партиции
  - `CURRENT ROW` — текущая строка
  - `n PRECEDING` — n строк назад
  - `n FOLLOWING` — n строк вперед

**Важно:** Если фрейм не указан явно, PostgreSQL использует фрейм по умолчанию:
```
RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
```
Это означает, что окно включает все строки от начала партиции до текущей строки. Для одних функций это поведение подходит, для других — требует явной корректировки.

Теперь, понимая, что такое фрейм, рассмотрим, как функции смещения взаимодействуют с ним.

## 1. Функции смещения: LAG и LEAD

Эти функции используются для доступа к строке, находящейся на заданное количество шагов *до* (`LAG`) или *после* (`LEAD`) текущей строки, без выполнения самообъединения (self-join) таблицы.

Важно отметить, что `LAG` и `LEAD` **не требуют определения фрейма**. Они работают исключительно на основе сортировки внутри партиции и всегда смотрят на строго заданное количество строк вперед или назад, игнорируя любые фреймы.

### Синтаксис
```sql
LAG(column [, offset [, default]]) OVER (PARTITION BY ... ORDER BY ...)
LEAD(column [, offset [, default]]) OVER (PARTITION BY ... ORDER BY ...)
```
*   `column` — значение, которое необходимо извлечь.
*   `offset` — смещение (по умолчанию 1). Положительное целое число.
*   `default` — значение по умолчанию, если смещение выходит за границы партиции (обычно NULL, если не указано).

### Пример 1. Разница во времени между событиями

Предположим, у нас есть таблица `user_actions` с данными о посещениях сайта.

```sql
-- Создание тестовых данных
CREATE TABLE user_actions (
    user_id INT,
    action_date DATE,
    page VARCHAR(50)
);

INSERT INTO user_actions VALUES
(1, '2023-01-10', 'Home'),
(1, '2023-01-12', 'Products'),
(1, '2023-01-15', 'Checkout'),
(2, '2023-01-11', 'Home'),
(2, '2023-01-14', 'Products');
```

**Задача:** Для каждого пользователя вычислить, сколько дней прошло с его предыдущего визита.

```sql
SELECT
    user_id,
    action_date,
    page,
    LAG(action_date, 1, action_date) OVER (PARTITION BY user_id ORDER BY action_date) AS previous_visit_date,
    action_date - LAG(action_date, 1, action_date) OVER (PARTITION BY user_id ORDER BY action_date) AS days_since_last_visit
FROM user_actions
ORDER BY user_id, action_date;
```

**Результат:**

| user_id | action_date | page     | previous_visit_date | days_since_last_visit |
|---------|-------------|----------|---------------------|-----------------------|
| 1       | 2023-01-10  | Home     | 2023-01-10          | 0                     |
| 1       | 2023-01-12  | Products | 2023-01-10          | 2                     |
| 1       | 2023-01-15  | Checkout | 2023-01-12          | 3                     |
| 2       | 2023-01-11  | Home     | 2023-01-11          | 0                     |
| 2       | 2023-01-14  | Products | 2023-01-11          | 3                     |

### Пример 2. Опережающее сравнение (LEAD)

**Задача:** Определить, является ли текущая цена акции выше, чем следующая (для анализа падений).

```sql
WITH stock_data AS (
    SELECT * FROM (VALUES
        ('AAPL', '2023-01-01', 150),
        ('AAPL', '2023-01-02', 152),
        ('AAPL', '2023-01-03', 151),
        ('AAPL', '2023-01-04', 153)
    ) AS t(ticker, date, price)
)
SELECT
    ticker,
    date,
    price,
    LEAD(price, 1) OVER (PARTITION BY ticker ORDER BY date) AS next_price,
    CASE
        WHEN price > LEAD(price) OVER (PARTITION BY ticker ORDER BY date) THEN 'Drop expected'
        ELSE 'Stable or Growth'
    END AS trend
FROM stock_data;
```

## 2. Функция FIRST_VALUE

`FIRST_VALUE` возвращает значение из первой строки окна. По умолчанию, если не указано иное, окно начинается с первой строки партиции и заканчивается текущей строкой. Это делает `FIRST_VALUE` предсказуемым и удобным для сравнения текущего значения с самым первым значением в группе.

### Синтаксис
```sql
FIRST_VALUE(column) OVER (PARTITION BY ... ORDER BY ...)
```

**Важно:** `FIRST_VALUE` по умолчанию использует фрейм `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`, что означает "от начала партиции до текущей строки". Это поведение нас устраивает, так как мы действительно хотим видеть самый первый элемент.

### Пример: Сравнение с первой покупкой

**Задача:** Показать стоимость заказа и то, на сколько текущая сумма отличается от первой покупки пользователя.

```sql
-- Данные о заказах
CREATE TABLE orders (
    user_id INT,
    order_date DATE,
    amount NUMERIC(10,2)
);

INSERT INTO orders VALUES
(1, '2023-01-01', 100),
(1, '2023-02-01', 150),
(1, '2023-03-01', 80),
(2, '2023-01-15', 200),
(2, '2023-02-15', 220);

SELECT
    user_id,
    order_date,
    amount,
    FIRST_VALUE(amount) OVER (PARTITION BY user_id ORDER BY order_date) AS first_purchase_amount,
    amount - FIRST_VALUE(amount) OVER (PARTITION BY user_id ORDER BY order_date) AS difference_from_first
FROM orders
ORDER BY user_id, order_date;
```

**Результат:**

| user_id | order_date | amount | first_purchase_amount | difference_from_first |
|---------|------------|--------|----------------------|-----------------------|
| 1       | 2023-01-01 | 100    | 100                  | 0                     |
| 1       | 2023-02-01 | 150    | 100                  | 50                    |
| 1       | 2023-03-01 | 80     | 100                  | -20                   |
| 2       | 2023-01-15 | 200    | 200                  | 0                     |
| 2       | 2023-02-15 | 220    | 200                  | 20                    |

## 3. Функция LAST_VALUE

`LAST_VALUE` возвращает значение из последней строки окна. Здесь кроется важная особенность, которая часто приводит к неожиданным результатам.

### Проблема: поведение по умолчанию

По умолчанию, если не указан фрейм, PostgreSQL использует:
```
RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
```

Это означает, что окно для `LAST_VALUE` заканчивается **на текущей строке**. Следовательно, "последняя строка окна" — это всегда текущая строка. В результате `LAST_VALUE` без явного фрейма ведет себя так же, как просто обращение к текущему значению столбца.

**Пример некорректного использования (дефолтное поведение):**

```sql
SELECT
    user_id,
    order_date,
    amount,
    LAST_VALUE(amount) OVER (PARTITION BY user_id ORDER BY order_date) AS last_amount_incorrect
FROM orders
ORDER BY user_id, order_date;
```

**Результат:**

| user_id | order_date | amount | last_amount_incorrect |
|---------|------------|--------|----------------------|
| 1       | 2023-01-01 | 100    | 100                  |
| 1       | 2023-02-01 | 150    | 150                  |
| 1       | 2023-03-01 | 80     | 80                   |
| 2       | 2023-01-15 | 200    | 200                  |
| 2       | 2023-02-15 | 220    | 220                  |

Как видите, столбец `last_amount_incorrect` просто дублирует `amount`, что делает использование `LAST_VALUE` бессмысленным.

### Решение: явное определение фрейма

Чтобы `LAST_VALUE` возвращал действительно последнюю строку во всей партиции, необходимо явно задать фрейм, который включает **все строки от начала до конца партиции**.

Фрейм `ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING` означает:
- `UNBOUNDED PRECEDING` — от самой первой строки партиции
- `UNBOUNDED FOLLOWING` — до самой последней строки партиции

**Корректный пример:**

```sql
SELECT
    user_id,
    order_date,
    amount,
    LAST_VALUE(amount) OVER (
        PARTITION BY user_id
        ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS last_amount_correct
FROM orders
ORDER BY user_id, order_date;
```

**Результат:**

| user_id | order_date | amount | last_amount_correct |
|---------|------------|--------|---------------------|
| 1       | 2023-01-01 | 100    | 80                  |
| 1       | 2023-02-01 | 150    | 80                  |
| 1       | 2023-03-01 | 80     | 80                  |
| 2       | 2023-01-15 | 200    | 220                 |
| 2       | 2023-02-15 | 220    | 220                 |

Теперь `last_amount_correct` показывает сумму последнего заказа для каждого пользователя на всех строках его заказов.

## 4. Функция NTH_VALUE

`NTH_VALUE` позволяет получить значение из *n-ной* строки в рамках окна. Как и `LAST_VALUE`, эта функция требует явного определения фрейма, если мы хотим получать значения из строк, находящихся после текущей.

### Синтаксис
```sql
NTH_VALUE(column, n) OVER (PARTITION BY ... ORDER BY ...)
```
Где `n` — номер строки (начиная с 1) в окне.

### Проблема: аналогичная LAST_VALUE

По умолчанию фрейм ограничен текущей строкой, поэтому если запросить `NTH_VALUE(..., 3)`, а текущая строка — первая или вторая, функция вернет NULL, так как третья строка еще не входит в окно.

### Решение: явное определение фрейма

Для корректной работы необходимо расширить фрейм до конца партиции.

**Задача:** Для каждого заказа пользователя вывести сумму его второго по счету заказа.

```sql
SELECT
    user_id,
    order_date,
    amount,
    NTH_VALUE(amount, 2) OVER (
        PARTITION BY user_id
        ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS second_order_amount
FROM orders
ORDER BY user_id, order_date;
```

**Результат:**

| user_id | order_date | amount | second_order_amount |
|---------|------------|--------|---------------------|
| 1       | 2023-01-01 | 100    | 150                 |
| 1       | 2023-02-01 | 150    | 150                 |
| 1       | 2023-03-01 | 80     | 150                 |
| 2       | 2023-01-15 | 200    | 220                 |
| 2       | 2023-02-15 | 220    | 220                 |

**Важное замечание:**
Если `n` не существует (например, у пользователя только один заказ, а мы запрашиваем `NTH_VALUE(..., 2)`), функция вернет `NULL`.

## 5. Почему LAG/LEAD не требуют фреймов, а LAST_VALUE/NTH_VALUE требуют?

Это ключевой момент для понимания различий между этими группами функций.

| Функция | Тип доступа | Необходимость фрейма | Причина |
|---------|-------------|---------------------|---------|
| `LAG` / `LEAD` | Смещение относительно текущей строки | **Не требуется** | Всегда смотрят на фиксированное количество строк вперед/назад, независимо от фрейма |
| `FIRST_VALUE` | Первая строка окна | **Не требуется явно** | Фрейм по умолчанию (от начала до текущей) дает нужный результат |
| `LAST_VALUE` | Последняя строка окна | **Требуется явно** | Фрейм по умолчанию заканчивается на текущей строке, что делает функцию бесполезной |
| `NTH_VALUE` | Произвольная позиция | **Требуется явно** | Аналогично `LAST_VALUE`, для доступа к строкам после текущей нужен расширенный фрейм |

### Правило запоминания:
- **`LAG` и `LEAD`** — работают "как есть", фрейм игнорируется.
- **`FIRST_VALUE`** — работает корректно без фрейма.
- **`LAST_VALUE` и `NTH_VALUE`** — **всегда** требуют явного указания фрейма `ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING`, если вы хотите получить значение из конца партиции или из строк, находящихся после текущей.

## 6. Сравнение производительности: Оконные функции vs Self-Join

Использование `LAG`, `LEAD` и подобных функций практически всегда значительно быстрее, чем самообъединение (self-join) таблицы для доступа к соседним записям.

**Почему:**
1.  **Однопроходный алгоритм:** Планировщик PostgreSQL может выполнить сортировку (или использовать индекс) один раз, а затем пройти по отсортированным данным, сохраняя значения предыдущих строк в памяти (или в рабочем файле).
2.  **Self-Join:** Требует как минимум двух сканирований таблицы (или двух обращений к индексу) и сложного условия соединения, что часто приводит к декартовому произведению в рамках групп, если не настроены индексы идеально.

### Пример сравнения логики:

**Self-Join (неоптимально):**
```sql
SELECT a.user_id, a.action_date, MAX(b.action_date)
FROM user_actions a
LEFT JOIN user_actions b ON a.user_id = b.user_id AND b.action_date < a.action_date
GROUP BY a.user_id, a.action_date;
```

**Оконная функция (оптимально):**
```sql
SELECT user_id, action_date,
       LAG(action_date) OVER (PARTITION BY user_id ORDER BY action_date)
FROM user_actions;
```
Второй вариант не только короче, но и выполняется быстрее за счет того, что не создает промежуточного расширенного набора строк (exploded rows).

## Заключение

Функции смещения и доступа к границам (`LAG`, `LEAD`, `FIRST_VALUE`, `LAST_VALUE`, `NTH_VALUE`) в PostgreSQL предоставляют мощный инструментарий для:

1.  **Анализа временных рядов** — разницы между датами, темпы прироста, сравнение с предыдущими периодами.
2.  **Сравнения строк в рамках одной группы** — сравнение с лидером, с аутсайдером, с первой или последней записью.
3.  **Замены сложных коррелированных подзапросов** и самообъединений.

### Ключевые выводы:
- **Фрейм** — это набор строк внутри партиции, который определяет, какие строки видит оконная функция. Понимание фрейма критически важно для корректной работы `LAST_VALUE` и `NTH_VALUE`.
- **`LAG` и `LEAD`** — идеальны для доступа к соседним строкам, не требуют фреймов.
- **`FIRST_VALUE`** — работает "из коробки" для сравнения с первым элементом.
- **`LAST_VALUE` и `NTH_VALUE`** — требуют явного указания фрейма `ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING` для корректной работы с концом партиции.
- Все оконные функции вычисляются после `WHERE`, `GROUP BY` и `HAVING`, но до `ORDER BY` основного запроса.
